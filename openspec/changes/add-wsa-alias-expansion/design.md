## Context

当前项目需要在 Windows 环境中提供复杂命令缩写能力，入口为 `sh/wsha.bat`，配置文件为 `config/wsh-alias.txt`。现有问题是命令较长且参数组合容易出错，且希望在保持批处理可用性的前提下兼容 Git Bash 调用方式。

约束与前提：
- 配置文件按行声明别名映射，支持注释与空行。
- 运行方式为 `wsa <alias> [args...]`，需要将用户参数正确转发到目标命令。
- 需要提供可诊断的错误输出与非零退出码，便于脚本链路使用。

## Goals / Non-Goals

**Goals:**
- 在 `wsha.bat` 中实现稳定的 alias 解析与命令展开。
- 定义清晰的参数合并规则：支持默认尾部追加和 `--` 占位插入。
- 对未知别名、非法配置、空输入提供明确错误反馈。
- 保持实现轻量，不引入第三方依赖。

**Non-Goals:**
- 不实现多级别名递归展开。
- 不实现带引号的复杂 shell 语法解析器（仅支持按空白分词）。
- 不覆盖跨平台（Linux/macOS）入口脚本。

## Decisions

1. 配置解析采用“逐行读取 + 首字段为 alias，剩余字段为模板命令”的模型。  
理由：与批处理能力匹配，实现成本低，可读性高。  
备选方案：使用 JSON/YAML 配置；未采用原因是引入额外解析依赖且不符合当前脚本生态。

2. 参数合并规则固定为两种模式：  
- 模板中包含 `--`：将运行时参数插入到 `--` 所在位置；模板其余参数保留。  
- 模板中不包含 `--`：运行时参数追加到模板末尾。  
理由：规则简单可预测，能覆盖示例中的前置/后置默认参数需求。  
备选方案：支持多个占位符或命名占位符；未采用原因是复杂度高且需求未提出。

3. 错误处理统一到标准错误输出并返回非零码。  
理由：调用方可通过 `ERRORLEVEL` 判断失败并在 CI/脚本中处理。  
备选方案：静默失败或仅提示不退出；未采用原因是会放大定位成本。

4. 文件路径基于仓库相对路径解析（相对脚本位置定位 `../config/wsh-alias.txt`）。  
理由：避免受当前工作目录影响，提升可移植性与稳定性。  
备选方案：依赖当前目录；未采用原因是易受调用入口变化影响。

## Risks / Trade-offs

- [批处理参数与转义边界复杂] -> 先覆盖无引号复杂转义的主路径，针对特殊字符场景增加回归测试与文档限制说明。  
- [配置书写错误导致行为异常] -> 在加载阶段做基本校验（别名为空、目标命令为空、重复别名），并输出行号。  
- [Windows 命令解析差异] -> 在设计中限制语法，明确“按空白分词”语义，避免承诺完整 shell 兼容。
